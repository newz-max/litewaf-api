## ADDED Requirements

### Requirement: Gateway routes requests by Host
The OpenResty gateway SHALL select the configured site by HTTP Host and proxy matching requests to that site's upstream.

#### Scenario: Matching host proxies upstream
- **WHEN** a request reaches the gateway with a Host matching a published site
- **THEN** the gateway proxies the request to the configured upstream and returns the upstream response when no blocking rule applies

#### Scenario: Unknown host is rejected
- **WHEN** a request reaches the gateway with a Host not present in the published configuration
- **THEN** the gateway returns an error response instead of proxying to an arbitrary upstream

### Requirement: Gateway loads local published configuration
The OpenResty gateway SHALL load WAF site, policy, and rule configuration from a local file generated by the publish process.

#### Scenario: Published config is loaded
- **WHEN** the gateway starts or reloads after a successful publish
- **THEN** it uses the published configuration for routing and rule enforcement

### Requirement: Gateway inspects request parameters
The gateway SHALL inspect configured request targets for enabled rules before proxying to upstream.

#### Scenario: Query parameter matches block rule
- **WHEN** a request query parameter matches an enabled `block` rule for the site policy
- **THEN** the gateway returns HTTP 403 without sending the request to upstream

#### Scenario: Query parameter matches log-only rule
- **WHEN** a request query parameter matches an enabled `log-only` rule for the site policy
- **THEN** the gateway records the match and still proxies the request to upstream

#### Scenario: Query parameter matches pass rule
- **WHEN** a request query parameter matches an enabled `pass` rule for the site policy
- **THEN** the gateway allows the request to continue without blocking due to that rule

### Requirement: Gateway emits structured WAF logs
The gateway SHALL emit JSON log entries for WAF rule matches containing site, rule, action, request, and client metadata available in the request context.

#### Scenario: Blocked request logs rule match
- **WHEN** the gateway blocks a request because of a matching rule
- **THEN** stdout contains a JSON log entry identifying the matched rule and action
