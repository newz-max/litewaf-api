# LiteWaf 贡献指南

> 语言 / Language: [中文](贡献指南.md) | [English](en/contributing.md)

感谢关注 LiteWaf。项目当前目标是保持源码开放、轻便、快速部署，贡献也应围绕这个方向：小步提交、真实验证、避免引入不必要的重量级依赖。

## 仓库边界

```text
.                         当前仓库，Go 控制面 API
deploy                    Compose 和部署脚本
doc                       文档
rules                     默认规则集
examples                  验证样例
../../openspec            工作区级规格和变更流程
litewaf-dashboard         配套前端仓库，Vue 3 + Naive UI 后台
litewaf-gateway           配套网关仓库，OpenResty 网关
```

## 本地检查

后端：

```bash
go test ./...
```

前端源码在配套仓库中维护：

```bash
npm install
npm run build
```

网关源码在配套仓库中维护，验证需要可用 Docker/OpenResty 环境：

```powershell
.\scripts\smoke.ps1 -Gateway http://localhost:18081 -HostHeader example.local
```

## 提交约定

- 提交应聚焦一个主题。
- 默认提交信息使用中文。
- 不提交 `node_modules/`、`dist/`、日志文件、`.env` 或密钥。
- 修改 API、规则字段、部署脚本或网关配置时，同步更新文档和验证说明。
- 修改用户可见工作流时，优先更新 [文档索引](文档索引.md) 中列出的权威文档；新增独立文档前必须确认它有明确读者、生命周期和导航入口。
- 不在业务页面或 API 中写 mock 数据；接口未接入时使用 loading、empty 或 error 状态。

## 安全配置

生产环境必须替换：

- `AUTH_TOKEN_SECRET`
- `GATEWAY_INGESTION_TOKEN`
- `LITEWAF_ADMIN_PASSWORD`
- `POSTGRES_PASSWORD`

不要在 issue、PR、文档样例以外的文件中提交真实密钥、连接串、备份包或生产 `.env`。

## 规则贡献

规则贡献应包含：

- 规则名称、类型、目标、动作、表达式、分数和启用默认值。
- 检测意图和误报风险说明。
- 至少一个本地验证请求。
- 是否建议默认 `block` 或先 `log-only`。

默认规则集要保持小而可审查。大规模托管规则、规则市场和插件机制将在后续阶段设计。
