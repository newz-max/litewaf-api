# OpenResty WAF 技术选型大纲

## 项目目标

本项目目标是构建一个可在 Docker 中运行的高性能 WAF 防护系统。整体设计采用数据面与控制面分离的架构：

- 数据面负责实时流量检测、拦截、限流和日志上报，要求低延迟、高吞吐、稳定可回滚。
- 控制面负责防护应用管理、证书管理、规则管理、策略发布、审计和可视化分析，要求易维护、可扩展、可运维。

## 总体架构

```text
Client
  |
  v
OpenResty WAF Gateway
  |-- IP / URI / UA 黑白名单
  |-- 请求频率限制
  |-- 并发连接限制
  |-- Header / Cookie / Args / Body 检测
  |-- SQLi / XSS / RCE / CC / Bot 规则
  |-- block / pass / log-only / rate-limit 动作
  |
  v
Upstream Service

Control Plane
  |
  |-- Go API
  |-- PostgreSQL
  |-- Redis / etcd
  |-- Dashboard
  |
  v
OpenResty worker 定时热加载规则

Log Pipeline
  |
  |-- stdout / Vector / Fluent Bit
  |-- Kafka 可选
  |-- ClickHouse 可选
  |
  v
报表 / 告警 / 攻击分析
```

## 技术栈定版

| 模块 | 技术选择 | 说明 |
| --- | --- | --- |
| WAF 网关 | OpenResty + Nginx + LuaJIT | 请求实时检测和拦截，运行在数据面 |
| WAF 规则执行 | Lua + lua-resty-core + ngx.re | 热路径尽量使用 LuaJIT 和 OpenResty 内建能力 |
| 限流控制 | lua-resty-limit-traffic | 支持请求速率限制、并发连接限制和组合限流 |
| 本地缓存 | lua_shared_dict + resty.lrucache | 存储规则版本、黑白名单、限流状态和临时风险分 |
| 控制面 API | Go | 负责规则、防护应用、证书、策略、发布、审计等后端能力 |
| 后台页面 | Vue 3 + TypeScript + Vite + Naive UI | 中后台管理页面，支持规则配置和日志查看 |
| 前端状态管理 | Pinia | 管理用户、菜单、防护应用、策略等前端状态 |
| 前端路由 | Vue Router | 后台页面路由管理 |
| 图表 | ECharts / vue-echarts | 仪表盘、攻击趋势、规则命中统计 |
| 请求库 | Axios | 调用控制面 API |
| 配置同步 | Redis / etcd | 规则热更新、策略版本和轻量状态同步 |
| 关系数据库 | PostgreSQL | 存储用户、租户、防护应用、监听、证书、规则、策略、发布记录 |
| 日志分析 | ClickHouse | 高吞吐 WAF 日志分析，第二阶段可加入 |
| 日志采集 | stdout + Vector / Fluent Bit | Docker 场景下优先输出 JSON 日志 |
| 部署 | Debian 12 minimal + Docker Compose 起步，兼容主流 Linux，后续 Kubernetes | 推荐 Debian 12 minimal 作为默认宿主机基线，但安装流程只依赖 Docker Compose |
| 压测 | wrk / vegeta / k6 | 验证 QPS、P95/P99 延迟和规则性能损耗 |

## Docker 服务划分

默认部署目标以 Debian 12 minimal 服务器为主，但不得依赖 Debian 专有宿主机能力。所有核心服务优先以 Docker 容器运行，宿主机只负责 Docker Engine、Docker Compose、系统资源限制、日志目录和防火墙等基础运行环境。其他主流 Linux 发行版作为兼容部署目标。

生产安装流程应学习成熟 WAF 的快速部署模型：安装脚本只做环境检查、下载 compose、生成 `.env`、拉取预构建镜像和启动容器。Go 编译、前端构建、OpenResty/Lua 依赖固定和镜像构建应由 CI 提前完成，避免在用户宿主机现场构建。

```text
waf-gateway
  OpenResty + LuaJIT
  处理真实业务流量

waf-api
  Go API
  提供控制面接口

waf-dashboard
  Vue 3 + Naive UI
  后台管理页面

postgres
  存储规则、策略、用户、审计、发布记录

redis / etcd
  配置热更新和轻量状态同步

clickhouse
  WAF 日志分析，可第二阶段加入

vector / fluent-bit
  日志采集和转发，可第二阶段加入
```

## OpenResty 数据面设计

### Nginx/Lua 阶段分工

| 阶段 | 职责 |
| --- | --- |
| init_by_lua | 加载 Lua 模块、初始化全局只读配置 |
| init_worker_by_lua | 定时拉取规则、刷新本地缓存、上报心跳 |
| rewrite_by_lua | URI 归一化、早期放行、基础规则判断 |
| access_by_lua | WAF 主检测、黑白名单、限流、风险评分和拦截 |
| header_filter_by_lua | 响应头安全策略 |
| body_filter_by_lua | 响应体检测，默认慎用 |
| log_by_lua | 异步日志、命中记录、限流连接离开统计 |

### WAF 检测对象

- Client IP
- Host
- URI
- Method
- Query args
- Headers
- Cookies
- Request body
- User-Agent
- Content-Type
- 上传文件元信息

### WAF 动作

- pass
- block
- log-only
- rate-limit
- temporary-ban
- captcha，可后续扩展

## 规则引擎设计

第一阶段建议使用 Lua 表描述规则，避免一开始引入过重的 DSL。

```lua
{
  id = "sqli-001",
  name = "SQLi union select",
  phase = "args",
  target = "query",
  action = "block",
  pattern = [[(?i)(union\s+select|sleep\(|benchmark\()]],
  score = 10,
  tags = { "sqli", "high" },
  enabled = true
}
```

后续可以演进为更完整的规则模型：

- 条件组合：AND / OR / NOT
- 匹配目标：uri / args / headers / cookies / body / ip / ua
- 匹配方式：regex / contains / equals / prefix / suffix / ip_cidr
- 处置动作：pass / log / block / rate-limit / temporary-ban
- 风险评分：多条规则累计 score，超过阈值再拦截
- 版本管理：规则集按版本发布，支持回滚

## Docker Compose MVP

Docker Compose MVP 推荐在 Debian 12 minimal 上验证，同时保持主流 Linux + Docker Compose 兼容。生产部署时需要提前配置 Docker Engine、Docker Compose、系统 ulimit、时区、日志轮转和开机自启动。

第一阶段可以先使用如下服务：

```yaml
services:
  waf-gateway:
    build: ./gateway
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./gateway/lua:/etc/waf/lua:ro
      - ./gateway/rules:/etc/waf/rules:ro
    depends_on:
      - redis

  waf-api:
    build: ./control-plane
    environment:
      DATABASE_URL: postgres://waf:waf@postgres:5432/waf
      REDIS_ADDR: redis:6379
    depends_on:
      - postgres
      - redis

  waf-dashboard:
    build: ./dashboard
    ports:
      - "8080:80"
    depends_on:
      - waf-api

  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: waf
      POSTGRES_PASSWORD: waf
      POSTGRES_DB: waf

  redis:
    image: redis:7-alpine
```

## 性能原则

- 请求热路径不访问远程数据库。
- 规则和策略提前同步到本地内存或共享字典。
- 黑白名单、静态资源、健康检查优先短路。
- 请求体检测按需开启，避免全量扫描。
- 正则规则需要缓存和分层匹配，避免每次请求动态构造复杂表达式。
- 日志使用异步投递或 stdout 输出，避免阻塞请求链路。
- 多租户策略需要按 Host、监听端口、协议或防护应用 ID 快速定位。
- 支持 fail-open / fail-close 配置。
- 每次规则变更都要可灰度、可审计、可回滚。

## 当前推荐结论

项目建议从以下组合起步：

```text
网关：OpenResty + LuaJIT
控制面：Go
后台：Vue 3 + TypeScript + Vite + Naive UI
存储：PostgreSQL + Redis
日志：JSON stdout 起步，ClickHouse 第二阶段加入
部署：Debian 12 minimal + Docker Compose 起步，兼容主流 Linux，后续 Kubernetes
```

这个组合可以兼顾性能、开发效率和后续扩展空间。第一阶段重点不是追求规则数量，而是先打通高性能请求链路、规则热更新、日志闭环、发布回滚和 Docker 化运行。生产安装体验以“几分钟拉起”为目标，依靠预构建镜像和 Compose 编排实现，而不是依靠宿主机现场构建。
