# LiteWaf 日志与可观测验证

本文档用于验证第 3 阶段日志与可观测能力。默认实现保持轻量：网关输出 JSON 日志到 stdout，并在配置 `LITEWAF_INGESTION_URL` 和 `LITEWAF_INGESTION_TOKEN` 后，以 best-effort 方式上报到 API；API 使用内存存储或 PostgreSQL 存储访问日志和 WAF 事件。

## English Guide

This guide validates LiteWaf logging and observability. The default implementation is lightweight: the Gateway writes JSON logs to stdout and, when `LITEWAF_INGESTION_URL` plus `LITEWAF_INGESTION_TOKEN` are configured, reports access logs and WAF events to the API on a best-effort basis. The API stores logs in memory or PostgreSQL depending on runtime configuration.

Key checks:

- Configure `GATEWAY_INGESTION_TOKEN`, `LITEWAF_INGESTION_URL`, metrics flags, sensitive header filtering, and value length limits.
- Generate normal proxied traffic and expected blocked/observed WAF traffic through the Gateway.
- Query access logs, attack logs, blocked/rejected records, observability summaries, protection overview, metrics, and dynamic-ban endpoints.
- Verify that sensitive values such as request bodies, Cookie, Authorization, dynamic tokens, captcha answers, signing secrets, and unbounded match values are not logged.
- For real client IP and GeoIP issues, validate trusted proxy CIDRs, forwarded headers, and access-log `client_ip` before interpreting country/region/city reports.

## 环境变量

```text
GATEWAY_INGESTION_TOKEN=change-me
METRICS_ENABLED=false
LITEWAF_METRICS_ENABLED=false
LITEWAF_SENSITIVE_HEADERS=authorization,cookie,set-cookie
LITEWAF_LOG_VALUE_MAX_LEN=160
LITEWAF_REAL_IP_TRUSTED_CIDRS=
LITEWAF_REAL_IP_HEADER=X-Forwarded-For
LITEWAF_REAL_IP_RECURSIVE=on
LITEWAF_DYNAMIC_BAN_CLEAR_INTERVAL=5
LITEWAF_GEOIP_DB_PATH=
LITEWAF_GEOIP_CACHE_SIZE=2048
```

`client_ip` 表示网关解析后的有效客户端 IP。直连部署下它等于连接来源地址；当网关前面存在受信任的 Docker bridge、负载均衡、CDN 或反向代理时，只有 `LITEWAF_REAL_IP_TRUSTED_CIDRS` 命中的来源才允许通过 `LITEWAF_REAL_IP_HEADER` 提供真实客户端 IP。`LITEWAF_REAL_IP_TRUSTED_CIDRS` 为空时，网关不会信任客户端伪造的 `X-Forwarded-For` 或 `X-Real-IP`。

统计报表地图不会信任请求头或日志 payload 中的地理字段。API 会在写入访问日志时基于有效 `client_ip` 查询 `LITEWAF_GEOIP_DB_PATH` 指向的本地 GeoIP/IP 网段 CSV；未配置、不可读、无匹配或私有/保留 IP 时只返回诊断，不生成假地区。部署时可通过 compose override 将宿主机 GeoIP 数据只读挂载到 API 容器，并设置 `LITEWAF_GEOIP_DB_PATH` 为容器内路径。

如果站点经过 Cloudflare、其他 CDN、宿主机 Nginx 或宝塔反向代理，GeoIP 地图显示的国家或城市明显不符合用户来源，优先检查 `access_logs.client_ip`。当 `client_ip` 落在 Cloudflare 网段，例如 `172.64.0.0/13`、`162.158.0.0/15`、`108.162.192.0/18`、`104.16.0.0/13` 等，说明 LiteWaf 记录的是 CDN 节点 IP，不是真实用户 IP。此时 GeoIP 数据本身可能是正常的，问题应在代理链路中恢复真实 IP。

Cloudflare 常见链路为：

```text
client -> Cloudflare -> host Nginx / BaoTa -> LiteWaf Gateway -> LiteWaf API
```

推荐处理方式是宿主机 Nginx 先信任 Cloudflare 官方 IP 段，并从 `CF-Connecting-IP` 恢复 `$remote_addr`；随后 Nginx 用 `X-Real-IP` 和 `X-Forwarded-For` 把恢复后的地址传给 LiteWaf Gateway。Gateway 再通过 `LITEWAF_REAL_IP_TRUSTED_CIDRS` 信任直接连接它的宿主机代理或 Docker bridge 地址，并从 `X-Forwarded-For` 读取有效客户端 IP。完整配置示例见 [Debian 12 minimal 部署说明](Debian12部署说明.md#cloudflare--cdn-前置时恢复真实客户端-ip)。

高级防护事件会复用同一条 WAF 事件日志链路。`score-threshold`、`body-inspection`、`upload-inspection` 和 `dynamic-ban` 事件只保存有限摘要、元数据、评分和封禁信息，不保存完整请求体。

`dynamic-ban` 事件还会驱动 API 侧活动封禁投影。网关上报封禁创建、刷新、执行或过期证据后，API 记录防护应用、监听端口、协议、有效客户端 IP、封禁原因、来源事件、创建时间、过期时间、剩余秒数、状态和 revision；手动释放会生成独立 clear revision，供各网关实例用摄取令牌增量拉取并清理本地 shared dict。`LITEWAF_DYNAMIC_BAN_CLEAR_INTERVAL` 控制网关 worker 拉取释放状态的间隔，默认 5 秒；请求热路径只读取本地状态，不调用 API。

CC 防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=cc-protection`、`category=rate-limit`、`rule_name`、`counter`、`threshold`、`window_sec`、`action` 和 `disposition`，用于解释 URL 频率限制、登录防爆破和 API 限流的处置原因。

攻击防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=attack-protection`、`category=managed`、`attack_type`、`group_name`、`rule_id`、`rule_name`、`target`、`action`、`score`、`summary` 和 `disposition`，用于解释 SQL 注入、XSS、RCE 和路径穿越托管规则的命中原因。`summary` 和 `normalized_value` 仍受 `LITEWAF_LOG_VALUE_MAX_LEN` 限制，不保存完整敏感 payload。

IP 黑白名单事件同样复用 WAF 事件日志链路。命中后会包含 `module=ip-access-list`、`category=ip-access-list`、`ip_access_list_id`、`ip_list_kind`、`ip_list_target`、`client_ip`、`action` 和 `disposition`，用于解释白名单放行或黑名单阻断结果。Exact IP 命中来自预处理哈希索引，CIDR 命中来自按前缀长度分组的索引；日志只保存有限名单元数据和有效客户端 IP。

访问控制事件同样复用 WAF 事件日志链路。命中后会包含 `module=access-control`、`category=access-control`、`rule_id`、`rule_name`、`target`、`action` 和 `disposition`，用于解释路径、Header 和 Host 访问控制规则的允许、观察或阻断结果。Header 命中日志只保存 Header 名称和有限摘要，不保存完整敏感 Header 值；Host 和路径摘要同样受长度限制。

上传防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=upload-protection`、`category=upload`、`rule_id`、`rule_name`、`target`、`action`、`disposition`、`threshold` 和 `upload_metadata`，用于解释危险扩展名、上传大小限制和观察模式。上传防护日志只保存文件名、扩展名、MIME、Content-Length 等有限元数据，不保存文件内容。

Bot / 人机验证事件同样复用 WAF 事件日志链路。命中后会包含 `module=bot-protection`、`category=challenge`、`rule_id`、`rule_name`、`target`、`action`、`disposition`、`challenge_mode`、`challenge_result`、`bot_result`、`bot_reason` 和 `device_signal`，用于解释 JavaScript challenge、本地 captcha、行为评分、设备信号和搜索引擎绕过结果。日志不得保存原始 challenge cookie、captcha 答案、签名密钥、Authorization、Cookie 或完整请求体。

动态防护 / 等候室事件同样复用 WAF 事件日志链路。命中后会包含 `module=dynamic-protection`、`category`、`rule_id`、`rule_name`、`target`、`action`、`disposition` 和 `advanced_target`，其中 `advanced_target` 用作动态结果字段。常见结果包括 `token-issued`、`token-passed`、`token-failed`、`mutation-applied`、`mutation-skipped`、`queue-admitted`、`queue-queued`、`queue-blocked` 和 `queue-observed`。日志不得保存原始动态 token、cookie 值、签名密钥、Authorization、Cookie 或完整 HTML 响应体。

Docker Compose 中网关默认使用：

```text
LITEWAF_INGESTION_URL=http://waf-api:8080
LITEWAF_INGESTION_TOKEN=${GATEWAY_INGESTION_TOKEN}
```

`DATABASE_URL` 为空时，API 使用进程内存保存日志，适合本地开发；Docker Compose 默认使用 PostgreSQL，日志表随 `db/init/001_mvp_schema.sql` 初始化。

## 手动验证

启动服务：

```bash
cd deploy
docker compose up -d --build
```

按 [使用说明](使用说明.md) 创建防护应用、配置策略并发布后，发送正常请求和攻击请求：

```bash
curl -i -H "Host: example.local" http://localhost:18081/
curl -i -H "Host: example.local" "http://localhost:18081/?q=union%20select"
```

检查网关 stdout：

```bash
docker compose logs gateway
```

预期看到 `event=access_log` 的 JSON 日志；命中规则、黑白名单、访问控制或限流时，预期看到 `event=waf_event` 的 JSON 日志。日志应包含 `request_id`，并且不包含 authorization、cookie 或请求体。

真实客户端 IP 验证可以在可用 Docker/OpenResty 环境中运行：

```powershell
cd ..\litewaf-gateway
pwsh .\scripts\real-ip-smoke.ps1
```

脚本会同时启动未配置受信代理的网关和配置受信 Docker 网段的网关，验证直连回退、可信 `X-Forwarded-For`、伪造头忽略、多跳转发链、访问控制、CC `client_ip`、CC `client_ip_path` 和临时封禁日志中的 `client_ip`。

生产环境排查 Cloudflare / CDN 真实 IP 时，可直接查询最新访问日志：

```bash
cd /opt/litewaf
docker exec litewaf-postgres-1 psql -U litewaf -d litewaf -Atc \
  "select host,client_ip,geo_country,geo_region,geo_city,created_at from access_logs order by id desc limit 20;"
```

若修复前看到 `client_ip` 是 Cloudflare 节点 IP，修复后新日志应变成真实公网 IP，并且 `geo_country`、`geo_region`、`geo_city` 会按真实 IP 重新解析。旧日志不会自动变更，因为 LiteWaf 默认不保存原始 `CF-Connecting-IP` 请求头；除非上游日志中另有记录，否则无法从已入库的 Cloudflare 节点 IP 反推出历史真实用户 IP。

## API 查询

登录获取 token 后查询日志：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/access-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/blocked-rejected-records"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/observability/summary"
```

可用过滤参数：

- 访问日志：`since`、`until`、`application_id`、`listener_port`、`scheme`、`host`、`client_ip`、`method`、`uri`、`status`、`disposition`
- 拦截/拒绝记录：`since`、`until`、`application_id`、`listener_port`、`scheme`、`host`、`client_ip`、`method`、`uri`、`status`、`disposition`、`module`、`action`、`trigger_source`
- 攻击日志：`since`、`until`、`application_id`、`listener_port`、`scheme`、`host`、`client_ip`、`rule_id`、`action`、`disposition`、`event_type`、`module`、`attack_type`、`challenge_result`、`bot_result`、`dynamic_result`
- 高级防护过滤：攻击日志额外支持 `advanced_target` 和 `min_score`

拦截/拒绝记录查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/blocked-rejected-records?disposition=blocked"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/blocked-rejected-records?trigger_source=access-log"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/blocked-rejected-records?module=attack-protection&action=block"
```

拦截/拒绝记录从访问日志中的 `blocked` / `rejected` 行开始，优先用同 `request_id` 的 WAF 事件解释触发模块和规则；没有 WAF 事件时，再展示动态封禁上下文或访问日志中的 `reason_code` / `reason`。如果没有任何解释证据，`trigger_source` 显示为 `unclassified`，不伪造模块、规则或封禁原因。未知 Host 等 access-only 网关拒绝会通过访问日志原因展示。

攻击防护查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=attack-protection&attack_type=sqli"
```

访问控制查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=access-control&action=block"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=access-control&action=log-only&disposition=observed"
```

IP 黑白名单查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=ip-access-list&action=block"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=ip-access-list&action=allow"
```

上传防护查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=upload-protection&action=block"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=upload-protection&action=log-only&disposition=observed"
```

Bot / 人机验证查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=issued"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=failed&disposition=blocked"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=passed"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&bot_result=captcha-failed"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&bot_result=search-engine-bypass"
```

动态防护 / 等候室查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=token-issued"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=token-failed&disposition=blocked"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=mutation-applied"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=queue-queued"
```

活动动态封禁查询和手动解封示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/dynamic-bans?application_id=1&listener_port=443&scheme=https"
curl -X POST "http://localhost:18080/api/v1/dynamic-bans/unban" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"application_id":1,"client_ip":"203.0.113.41"}'
```

手动解封会写入 `resource_type=dynamic_ban`、`action=unban` 的审计日志，`resource_id` 使用 `<application_id>:<client_ip>`。成功、no-op 和认证后失败都会保留 actor、role、目标、结果和时间；审计记录不保存原始请求体、Cookie、Authorization、网关摄取令牌或动态 token。网关同步到释放 revision 后会输出 `event=dynamic_ban_clear` 的有界 JSON 日志，用于确认目标防护应用/IP 已从本地动态封禁字典删除；同一 IP 在其他防护应用的封禁不受影响，后续新违规仍可重新触发封禁。

`/api/v1/observability/summary` 的 `attack_protection` 字段按 `attack_type|action|disposition` 返回计数，例如 `sqli|block|blocked` 或 `xss|log-only|observed`。

`/api/v1/observability/summary` 的 `access_control` 字段按 `action|disposition` 返回计数，例如 `block|blocked`、`log-only|observed` 或 `allow|allowed`。存在访问控制事件时，Top 规则维度也可以包含访问控制规则名。

`/api/v1/observability/summary` 的 `upload_protection` 字段按 `action|disposition` 返回计数，例如 `block|blocked` 或 `log-only|observed`。存在上传防护事件时，Top 规则维度也可以包含上传防护规则名。

`/api/v1/observability/summary` 的 `bot_protection` 字段按 `challenge_result|bot_result|action|disposition` 返回计数，例如 `issued|captcha-issued|block|blocked`、`passed|behavior-pass|pass|proxied`、`passed|search-engine-bypass|pass|proxied` 或 `failed|captcha-failed|block|blocked`。存在 Bot 验证事件时，Top 规则维度也可以包含 Bot 验证规则名。

`/api/v1/observability/summary` 的 `dynamic_protection` 字段按 `category|result|action|disposition` 返回计数，例如 `dynamic-token|token-issued|block|proxied`、`dynamic-token|token-failed|block|blocked`、`page-mutation|mutation-applied|log-only|proxied` 或 `waiting-room|queue-queued|waiting-room|blocked`。存在动态防护事件时，Top 规则维度也可以包含动态防护规则名。

无匹配数据时接口应返回成功响应和空数组。

## Dashboard 验证

进入 Dashboard 后检查：

- 仪表盘显示请求总数、拦截/拒绝、WAF 命中、限流次数和 Top 列表。
- 拦截/拒绝记录页面可按防护应用、监听、Host、IP、URI、状态、处置、模块、动作、触发来源和时间范围过滤。
- 拦截/拒绝记录页面可显示 WAF 事件、动态封禁、访问日志原因或未归类来源，并只在有真实上下文时提供攻击日志、动态封禁和模块回跳。
- 攻击日志页面可按防护应用、监听、IP、规则、动作、处置、事件类型、模块和攻击类型过滤。
- 攻击日志页面可按 Bot challenge 结果和 Bot 增强结果过滤。
- 攻击日志页面可按动态防护结果过滤。
- 攻击日志页面可显示评分、高级检测目标、Body 元数据、上传元数据、动态封禁原因、CC 防护解释字段、攻击防护字段、访问控制字段、上传防护字段、Bot 验证/增强字段和动态防护字段。
- 动态封禁页面显示真实活动封禁、剩余时长、状态和解封结果；readonly 和 auditor 用户只能查看，管理员解封时使用页面内确认流程。
- 访问日志页面可按防护应用、监听、Host、IP、方法、URI、状态码和处置过滤。
- 无数据时页面显示空态，不显示 mock 数据。

## Metrics 验证

如需暴露指标，在 `.env` 中启用：

```text
METRICS_ENABLED=true
LITEWAF_METRICS_ENABLED=true
```

检查 API 指标：

```bash
curl http://localhost:18080/metrics
```

检查网关指标：

```bash
curl http://localhost:18081/metrics
```

指标端点默认关闭，生产环境建议仅在内网或受控网络中启用。后续可接入 Prometheus；Vector、Fluent Bit 和 ClickHouse 可作为日志采集与分析增强项，不是当前阶段的必需依赖。
