# LiteWaf 日志与可观测验证

本文档用于验证第 3 阶段日志与可观测能力。默认实现保持轻量：网关输出 JSON 日志到 stdout，并在配置 `LITEWAF_INGESTION_URL` 和 `LITEWAF_INGESTION_TOKEN` 后，以 best-effort 方式上报到 API；API 使用内存存储或 PostgreSQL 存储访问日志和 WAF 事件。

## 环境变量

```text
GATEWAY_INGESTION_TOKEN=change-me
METRICS_ENABLED=false
LITEWAF_METRICS_ENABLED=false
LITEWAF_SENSITIVE_HEADERS=authorization,cookie,set-cookie
LITEWAF_LOG_VALUE_MAX_LEN=160
```

高级防护事件会复用同一条 WAF 事件日志链路。`score-threshold`、`body-inspection`、`upload-inspection` 和 `dynamic-ban` 事件只保存有限摘要、元数据、评分和封禁信息，不保存完整请求体。

CC 防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=cc-protection`、`category=rate-limit`、`rule_name`、`counter`、`threshold`、`window_sec`、`action` 和 `disposition`，用于解释 URL 频率限制、登录防爆破和 API 限流的处置原因。

攻击防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=attack-protection`、`category=managed`、`attack_type`、`group_name`、`rule_id`、`rule_name`、`target`、`action`、`score`、`summary` 和 `disposition`，用于解释 SQL 注入、XSS、RCE 和路径穿越托管规则的命中原因。`summary` 和 `normalized_value` 仍受 `LITEWAF_LOG_VALUE_MAX_LEN` 限制，不保存完整敏感 payload。

访问控制事件同样复用 WAF 事件日志链路。命中后会包含 `module=access-control`、`category=access-control`、`rule_id`、`rule_name`、`target`、`action` 和 `disposition`，用于解释 IP/CIDR、路径、Header 和 Host 访问控制规则的允许、观察或阻断结果。Header 命中日志只保存 Header 名称和有限摘要，不保存完整敏感 Header 值；Host 和路径摘要同样受长度限制。

上传防护事件同样复用 WAF 事件日志链路。命中后会包含 `module=upload-protection`、`category=upload`、`rule_id`、`rule_name`、`target`、`action`、`disposition`、`threshold` 和 `upload_metadata`，用于解释危险扩展名、上传大小限制和观察模式。上传防护日志只保存文件名、扩展名、MIME、Content-Length 等有限元数据，不保存文件内容。

Bot / 人机验证事件同样复用 WAF 事件日志链路。命中后会包含 `module=bot-protection`、`category=challenge`、`rule_id`、`rule_name`、`target`、`action`、`disposition`、`challenge_mode` 和 `challenge_result`，用于解释 JavaScript challenge 的发放、通过、失败、阻断和观察结果。日志不得保存原始 challenge cookie、签名密钥、Authorization、Cookie 或完整请求体。

动态防护 / 等候室事件同样复用 WAF 事件日志链路。命中后会包含 `module=dynamic-protection`、`category`、`rule_id`、`rule_name`、`target`、`action`、`disposition` 和 `advanced_target`，其中 `advanced_target` 用作动态结果字段。常见结果包括 `token-issued`、`token-passed`、`token-failed`、`mutation-applied`、`mutation-skipped`、`queue-admitted`、`queue-queued`、`queue-blocked` 和 `queue-observed`。日志不得保存原始动态 token、cookie 值、签名密钥、Authorization、Cookie 或完整 HTML 响应体。

Docker Compose 中网关默认使用：

```text
LITEWAF_INGESTION_URL=http://waf-api:8080
LITEWAF_INGESTION_TOKEN=${GATEWAY_INGESTION_TOKEN}
```

`DATABASE_URL` 为空时，API 使用进程内存保存日志，适合本地开发；Docker Compose 默认使用 PostgreSQL，日志表随 `db/init/001_mvp_schema.sql` 初始化。

## 手动验证

启动服务：

```bash
cd deploy
docker compose up -d --build
```

按 `doc/MVP防护闭环验证.md` 创建站点、策略并发布后，发送正常请求和攻击请求：

```bash
curl -i -H "Host: example.local" http://localhost:18081/
curl -i -H "Host: example.local" "http://localhost:18081/?q=union%20select"
```

检查网关 stdout：

```bash
docker compose logs gateway
```

预期看到 `event=access_log` 的 JSON 日志；命中规则、黑白名单、访问控制或限流时，预期看到 `event=waf_event` 的 JSON 日志。日志应包含 `request_id`，并且不包含 authorization、cookie 或请求体。

## API 查询

登录获取 token 后查询日志：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/access-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/observability/summary"
```

可用过滤参数：

- 访问日志：`since`、`until`、`site_id`、`host`、`client_ip`、`method`、`uri`、`status`、`disposition`
- 攻击日志：`since`、`until`、`site_id`、`client_ip`、`rule_id`、`action`、`disposition`、`event_type`、`module`、`attack_type`、`challenge_result`、`dynamic_result`
- 高级防护过滤：攻击日志额外支持 `advanced_target` 和 `min_score`

攻击防护查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=attack-protection&attack_type=sqli"
```

访问控制查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=access-control&action=block"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=access-control&action=log-only&disposition=observed"
```

上传防护查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=upload-protection&action=block"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=upload-protection&action=log-only&disposition=observed"
```

Bot / 人机验证查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=issued"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=failed&disposition=blocked"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=bot-protection&challenge_result=passed"
```

动态防护 / 等候室查询示例：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=token-issued"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=token-failed&disposition=blocked"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=mutation-applied"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs?module=dynamic-protection&dynamic_result=queue-queued"
```

`/api/v1/observability/summary` 的 `attack_protection` 字段按 `attack_type|action|disposition` 返回计数，例如 `sqli|block|blocked` 或 `xss|log-only|observed`。

`/api/v1/observability/summary` 的 `access_control` 字段按 `action|disposition` 返回计数，例如 `block|blocked`、`log-only|observed` 或 `allow|allowed`。存在访问控制事件时，Top 规则维度也可以包含访问控制规则名。

`/api/v1/observability/summary` 的 `upload_protection` 字段按 `action|disposition` 返回计数，例如 `block|blocked` 或 `log-only|observed`。存在上传防护事件时，Top 规则维度也可以包含上传防护规则名。

`/api/v1/observability/summary` 的 `bot_protection` 字段按 `challenge_result|action|disposition` 返回计数，例如 `issued|block|blocked`、`passed|pass|proxied` 或 `failed|block|blocked`。存在 Bot 验证事件时，Top 规则维度也可以包含 Bot 验证规则名。

`/api/v1/observability/summary` 的 `dynamic_protection` 字段按 `category|result|action|disposition` 返回计数，例如 `dynamic-token|token-issued|block|proxied`、`dynamic-token|token-failed|block|blocked`、`page-mutation|mutation-applied|log-only|proxied` 或 `waiting-room|queue-queued|waiting-room|blocked`。存在动态防护事件时，Top 规则维度也可以包含动态防护规则名。

无匹配数据时接口应返回成功响应和空数组。

## Dashboard 验证

进入 Dashboard 后检查：

- 仪表盘显示请求总数、拦截/拒绝、WAF 命中、限流次数和 Top 列表。
- 攻击日志页面可按站点、IP、规则、动作、处置、事件类型、模块和攻击类型过滤。
- 攻击日志页面可按 Bot challenge 结果过滤。
- 攻击日志页面可按动态防护结果过滤。
- 攻击日志页面可显示评分、高级检测目标、Body 元数据、上传元数据、动态封禁原因、CC 防护解释字段、攻击防护字段、访问控制字段、上传防护字段、Bot 验证字段和动态防护字段。
- 访问日志页面可按站点、Host、IP、方法、URI、状态码和处置过滤。
- 无数据时页面显示空态，不显示 mock 数据。

## Metrics 验证

如需暴露指标，在 `.env` 中启用：

```text
METRICS_ENABLED=true
LITEWAF_METRICS_ENABLED=true
```

检查 API 指标：

```bash
curl http://localhost:18080/metrics
```

检查网关指标：

```bash
curl http://localhost:18081/metrics
```

指标端点默认关闭，生产环境建议仅在内网或受控网络中启用。后续可接入 Prometheus；Vector、Fluent Bit 和 ClickHouse 可作为日志采集与分析增强项，不是当前阶段的必需依赖。
