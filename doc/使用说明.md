# LiteWaf 使用说明

> 语言 / Language: [中文](使用说明.md) | [English](en/usage.md)

本文档面向已经启动 LiteWaf 的操作员，说明从创建防护应用到网关真正生效的完整流程。核心规则是：应用、监听、证书、规则、策略、IP 黑白名单、访问控制、限流和防护模块的写入只更新控制面数据；必须发布新版本后，OpenResty 网关才会加载并执行这些变更。

## 入口和权限

默认入口：

| 服务 | 地址 | 说明 |
| --- | --- | --- |
| Dashboard | `http://服务器IP:18080` | 后台管理页面 |
| Gateway | 应用发布的监听端口 | WAF 网关入口，默认由 Gateway 直接监听宿主机端口 |
| API | Dashboard `/api/` 反代 | 管理 API |

角色约定：

- `admin`：可创建、修改、删除、发布、回滚和查看审计。
- `auditor`：可查看配置和审计，不能写入或发布。
- `readonly`：只能查看配置。

后台页面和 API 无数据时显示空态或返回空数组，不使用 mock 行。首次部署后看到空应用、空日志或空发布记录是正常状态。

## 标准上线流程

1. 登录 Dashboard，进入“防护应用”创建应用，填写应用名称、域名、入口端口、HTTP/HTTPS 协议、证书、上游地址、运行模式和启用状态。
2. 配置防护策略、默认规则或模块规则。常见模块包括 CC 防护、攻击防护、IP 黑白名单、访问控制、上传防护、Bot / 人机验证、动态防护和高级规则生态。
3. 进入“发布记录”，点击“发布新版本”。后台会先调用发布预览，展示应用、监听矩阵、证书风险、规则、策略、模块汇总、兼容诊断和风险提示；确认后才生成新发布版本。
4. 确认应用域名解析到 Gateway 所在服务器，然后通过应用 Host、监听端口和协议验证正常代理请求和至少一个预期阻断或观察事件。HTTPS 入口需要在应用监听中绑定证书。
5. 查看攻击日志、访问日志、观测汇总和审计日志，确认网关行为、发布版本和操作记录一致。

如果只完成第 1 步或第 2 步但没有发布，网关仍继续使用上一版配置。如果域名和端口没有进入 Gateway，Dashboard 也不会出现对应访问日志；生产默认使用 host-network，让 Gateway 按发布配置监听 `80/http`、`443/https` 和自定义端口。开发 Compose 仍保留单端口 `18081` 便于本地验证。

## 防护应用创建和发布

防护应用可以通过 Dashboard“防护应用”或 API 创建：

```bash
curl -X POST http://localhost:18080/api/v1/applications \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Example",
    "mode": "protect",
    "enabled": true,
    "hosts": [{"host": "example.local", "is_primary": true}],
    "listeners": [
      {"port": 80, "protocol": "http", "enabled": true},
      {"port": 443, "protocol": "https", "certificate_id": 1, "enabled": true}
    ],
    "upstreams": [{"name": "primary", "url": "http://upstream:8080", "weight": 1, "enabled": true}]
  }'
```

创建、编辑、删除或启停应用、监听、证书绑定后，需要发布：

```bash
curl -X GET http://localhost:18080/api/v1/releases/preview \
  -H "Authorization: Bearer <token>"

curl -X POST http://localhost:18080/api/v1/releases \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"operator":"admin","note":"application update"}'
```

发布成功后，网关按 `Host + 监听端口 + 协议` 匹配已发布应用。未知 Host、端口或协议组合应返回 404，避免代理到任意上游。

发布还会在运行时目录生成 `listeners/body-size.conf`，写入当前网关请求体硬上限，例如 `client_max_body_size 50m;`。这个限制在请求进入 Lua/WAF 规则前由 OpenResty 生效，和上传防护规则不是同一层。

## 防护配置发布边界

以下变更都需要发布后才会被网关执行：

| 变更类型 | 后台入口或 API | 发布后效果 |
| --- | --- | --- |
| 应用新增、修改、删除、启停 | 防护应用 / `/api/v1/applications` | Host、端口、协议、证书、upstream 和防护模式更新 |
| 证书上传、删除或绑定 | 证书管理 / `/api/v1/certificates` | HTTPS 监听证书文件和证书风险更新 |
| 策略和默认规则 | 策略、规则管理 / `/api/v1/policies`、`/api/v1/rules` | 应用绑定规则、阈值和高级检测字段更新 |
| IP 黑白名单 | IP 黑白名单 / `/api/v1/ip-access-lists` | `ip_access_index` 更新，网关热路径按 Exact IP 或 CIDR 判断 |
| 访问控制 | 访问控制 / `/api/v1/access-control/rules` | 路径、Header、Host 允许、观察或阻断规则更新 |
| CC 防护和限流兼容规则 | CC 防护、限流兼容入口 | URL 频率、登录爆破、API 限流和封禁规则更新 |
| 上传防护 | 上传防护 / `/api/v1/upload-protection/rules` | 上传路径、扩展名和大小限制更新 |
| Bot / 人机验证 | Bot / 人机验证 / `/api/v1/bot-protection/rules` | JS challenge、captcha、行为评分和设备信号规则更新 |
| 动态防护 / 等候室 | 动态防护 / `/api/v1/dynamic-protection/rules` | 动态令牌、页面动态化和等候室规则更新 |
| 规则包导入或规则测试结果 | 高级规则生态 | 导入后的可执行规则和发布预览警告更新 |

模块页面里的“模拟预览”“规则测试”“远程包预览”“导出预览”只用于审查和风险判断，不会修改网关活动配置。实际执行仍以发布版本为准。

## 验证网关行为

正常代理请求：

```bash
curl -i -H "Host: example.local" http://localhost/
```

攻击样例：

```bash
curl -i -H "Host: example.local" "http://localhost/?q=union%20select"
curl -i -H "Host: example.local" "http://localhost/?q=%3Cscript%3Ealert(1)%3C/script%3E"
```

上传防护样例：

```bash
curl -i -X POST \
  -H "Host: example.local" \
  -F "file=@./doc/使用说明.md;filename=shell.php;type=application/octet-stream" \
  http://localhost/upload
```

未知 Host：

```bash
curl -i -H "Host: unknown.local" http://localhost/
```

预期结果：

- 正常 Host 和路径代理到上游。
- 命中阻断规则时返回 403 或模块定义的阻断/等候室响应。
- 命中观察规则时请求继续通过，但攻击日志记录 `disposition=observed`。
- 未发布的新应用、新监听或新规则不会影响当前网关行为。

## 日志、审计和观测

常用查询：

```bash
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/access-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/blocked-rejected-records"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/attack-logs"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/observability/summary"
curl -H "Authorization: Bearer <token>" "http://localhost:18080/api/v1/audit-logs"
```

排查被拦截或被拒绝的请求时，优先打开后台“日志审计 / 拦截 / 拒绝记录”。该页面从访问日志中的 `blocked` / `rejected` 请求出发，展示 Host、URI、来源 IP、状态码、处置结果、触发来源和原因，并在有证据时回跳到攻击日志、动态封禁或对应防护模块。

触发来源含义：

- `waf-event`：同请求 ID 存在 WAF 事件，可以看到模块、规则、动作和摘要。
- `dynamic-ban`：请求命中了动态封禁上下文。
- `access-log`：只有访问日志自带的网关原因，例如未知 Host 或访问路径在 WAF 事件前被拒绝。
- `unclassified`：目前只有访问日志，系统不会伪造攻击规则或封禁原因。

按模块排查：

```text
GET /api/v1/attack-logs?module=cc-protection
GET /api/v1/attack-logs?module=attack-protection&attack_type=sqli
GET /api/v1/attack-logs?module=ip-access-list&action=block
GET /api/v1/attack-logs?module=access-control&action=block
GET /api/v1/attack-logs?module=upload-protection&action=block
GET /api/v1/attack-logs?module=bot-protection&challenge_result=failed
GET /api/v1/attack-logs?module=dynamic-protection&dynamic_result=token-failed
```

无匹配数据时接口返回成功响应和空数组，Dashboard 展示空态。排障时先确认最近发布版本，再看拦截/拒绝记录、访问日志和攻击日志是否出现对应应用 ID、Host、监听端口、协议、模块和动作字段。

## 发布回滚

后台“发布记录”支持查看历史版本并回滚。API 回滚入口：

```bash
curl -X POST "http://localhost:18080/api/v1/releases/<version>/rollback" \
  -H "Authorization: Bearer <token>"
```

回滚会把历史发布配置、监听文件和证书引用重新写入 `GATEWAY_CONFIG_PATH` 所在运行时目录，并生成新的回滚发布记录和审计日志。回滚只恢复网关活动配置，不等同于撤销数据库里的应用、规则或策略编辑；后续再次发布会基于当前控制面数据生成新配置。

## 备份、恢复和升级

生产运维使用安装目录中的 `litewafctl.sh`：

```bash
cd /opt/litewaf
sudo ./litewafctl.sh health
sudo ./litewafctl.sh diagnose
sudo ./litewafctl.sh backup
sudo ./litewafctl.sh upgrade v1.0.1
sudo ./litewafctl.sh rollback
```

备份恢复、升级和镜像标签回滚的完整说明见 [Debian 12 minimal 部署说明](Debian12部署说明.md)。

## 常见问题

### 创建应用后网关仍返回 404

确认应用 Host、监听端口和协议与请求一致，并在“发布记录”发布新版本。只保存应用不会更新网关活动配置。

### 修改规则后没有拦截

检查规则是否启用、是否绑定到策略或对应模块，动作是否为 `log-only`，然后执行发布。再通过拦截/拒绝记录确认是否出现 `blocked` / `rejected`，通过攻击日志确认是否为观察事件而非阻断事件。

### 上传接口返回 413

413 可能来自网关请求体硬上限，而不是上传防护规则命中。先在“系统设置 / 运行信息”或“发布记录 / 发布预览”查看“网关请求体上限”，默认是 `50m`。生产主机可检查实际 OpenResty 配置和错误日志：

```bash
docker exec litewaf-gateway-1 /usr/local/openresty/bin/openresty -T | grep -n client_max_body_size
docker logs litewaf-gateway-1 2>&1 | grep -i "too large body"
```

如果 413 发生在 OpenResty 请求体读取阶段，当前不会产生上传防护 WAF 事件。需要调整 `.env` 中的 `LITEWAF_GATEWAY_CLIENT_MAX_BODY_SIZE`，重新安装/启动或发布后让 API 写出新的 `listeners/body-size.conf`。

### 发布预览出现风险提示

风险提示来自真实配置摘要，例如监听端口冲突、HTTPS 缺少证书、证书域名不匹配、bridge-range 超出映射范围、全局低阈值、宽泛路径、阻断类动作或未测试的阻断规则。建议先用观察模式或缩小应用/路径范围，验证后再发布阻断。

### 页面没有数据

后台使用真实 API 数据。首次启动、未创建应用、未发布配置或没有流量时，表格为空是正常状态。检查 API 错误提示、健康检查和服务日志。
